A/B testing is one of the most powerful tools a data scientist can wield. Done right, it lets you make causal claims from observational data at scale. Done wrong, it creates a false sense of certainty and ships broken products. This chapter builds the mental foundation you need before touching a single formula.

Core Idea

An A/B test is a randomized controlled experiment. You expose users randomly to two or more experiences and measure whether the difference in outcomes is real or just noise.

01 /

The Mental Model

Every user has two potential outcomes:

ITE = Y(1) - Y(0) // Individual Treatment Effect — permanently unobservable
Y(0) → what happens if the user sees control
Y(1) → what happens if the user sees treatment

// You can NEVER observe both for the same person.
// This is the Fundamental Problem of Causal Inference.

ATE = E[Y(1)] - E[Y(0)] // Average Treatment Effect (what A/B tests estimate)

// Why randomization works:
E[Y(0) | Treatment] = E[Y(0) | Control] // Control = valid counterfactual

Randomization solves this. When users are randomly assigned, the two groups are statistically identical on average. Any difference you observe is causally attributable to the treatment, not pre-existing user differences.

Why Randomization Matters

Without randomization you get correlation, not causation. Users who click "Try New Feature" are already more engaged. Comparing them to the rest gives you selection bias, not a treatment effect.

Key Assumptions

Five assumptions must hold for your ATE estimate to be valid. Violate any one and your results are biased.

01
SUTVA
No interference between users. No hidden treatment variations.
02
Positivity
Every user has non-zero probability of being in either arm.
03
Randomization
Assignment is independent of user characteristics.
04
No Peeking
Analysis only at pre-specified time. Not when results look good.
05
Stable Population
Users during the experiment represent the target population.

SUTVA breaks constantly in real products. When it does, your ATE estimate is biased downward — the true effect is larger than what you measured.

Company How SUTVA breaks
MetaTreatment user posts more → changes content seen by control users in their network
AirbnbDiscounts to treatment guests reduce listing availability for control guests
UberTreatment rider surge pulls drivers away from control riders
EmailControl user receives a forwarded promo from a treatment user

Hypothesis Testing

You never prove a hypothesis. You test against a null and decide whether to reject it.

H0 (null): The treatment has no effect.
Any observed difference is due to chance.

H1 (alternative): The treatment has a real effect.

// p-value = P(seeing this result | H0 is true)
// It is NOT P(H0 is true). A common and costly misread.
Common Misconception

A p-value of 0.03 does not mean there is a 3% chance the null is true. It means: if the null were true, you would see a result this extreme only 3% of the time. Subtle difference, massive consequences in interviews.

04 /

Type I & Type II Errors

Type I Error
False Positive

You reject H0 when it is actually true. You ship a change that had no real effect. Rate controlled by alpha (usually 0.05).

Type II Error
False Negative

You fail to reject H0 when it is false. You miss a real effect and don't ship something that works. Rate is beta (usually 0.20).

Statistical power = 1 - beta. A power of 0.80 means you catch 80% of real effects of the size you care about.

Industry Context

At companies like Meta or Google, Type I errors are usually more costly since shipping a bad change harms millions of users. But at high-velocity teams running hundreds of experiments, unchecked Type II errors mean real wins never ship.

05 /

When NOT to Run an A/B Test

This question comes up constantly in interviews. Interviewers want to see that you understand the limits of experimentation, not just its mechanics.

Scenario Why Testing Fails Alternative
Not enough traffic Cannot reach statistical power in a reasonable time window Quasi-experiments, observational analysis
Network effects Treatment leaks from treated users to control users (SUTVA violation) Cluster randomization, geo experiments
One-time events No reproducible population to randomize over Historical comparison, synthetic control
Irreversible changes Cannot safely run treatment on a subset (e.g., pricing model overhaul) Phased rollout with monitoring
Ethical constraints Cannot randomize access to safety, legal, or compliance features Natural experiments, diff-in-diff

Company Flavors

Each company has a distinct experimentation culture shaped by their product, scale, and constraints. Know these going into company-specific interviews.

Meta

Speed and scale. Ships fast. High bar for movement on engagement and ads metrics. Extensive use of holdouts for long-term effect measurement.

Google

Rigorous. Long ramp periods, conservative on user-facing changes. Dogfood extensively before external launch.

Airbnb

Two-sided marketplace complexity. Experiments must account for both host and guest populations. SUTVA violations are a core challenge.

Netflix

Long-term effects matter. Short-term engagement can conflict with 6-month retention. Heavy focus on qualitative + quantitative signals.

Uber

Network effects and interference are central. Pricing experiments are especially tricky because of equilibrium effects on supply and demand.

Microsoft

ExP platform pioneers. Invented CUPED for variance reduction. Ron Kohavi's team defined many industry norms. Strong on trustworthiness checks like SRM detection.

Amazon

Guardrail-heavy. Extremely conservative ship bar. Revenue and customer experience guardrails rarely negotiable even when primary metrics win.

Adobe

Sequential testing, causal inference, and multi-armed bandits via Adobe Target. Strong focus on variance reduction and early stopping methods.

07 /

Key Terms

ATE

Average Treatment Effect. The expected difference in outcomes between treatment and control across the full population.

MDE

Minimum Detectable Effect. The smallest effect size your experiment is powered to detect.

SRM

Sample Ratio Mismatch. When observed traffic split differs from intended split. A red flag that randomization broke.

SUTVA

Stable Unit Treatment Value Assumption. One user's treatment should not affect another user's outcome.

Guardrail Metric

A metric you must not harm, even if your primary metric improves. Protects user experience and business health.

HiPPO

Highest Paid Person's Opinion. When executive intuition overrides experimental evidence. A culture problem.

Power (1-β)

Probability of detecting a real effect when one exists. Industry standard is 0.80 (80% power).

p-value

P(observing this result | H0 is true). Not the probability the null is true. Not permission to ship.

08 /

Interview FAQ

These are the questions that separate strong candidates from everyone else. Each one has a trap. Know the trap before you answer.

Our test showed p = 0.04. Should we ship? +
Trap: Answering yes immediately

The right answer walks through five checks before saying ship:

1. Is the effect practically significant, not just statistically? A 0.001% lift with p=0.04 is meaningless at scale unless you're Google Search.
2. Did any guardrail metrics degrade? A CTR win that hurts session length is often not worth shipping.
3. Was there an SRM? If traffic split was off, the p-value is unreliable.
4. Was the test run long enough? Did you avoid novelty effect by running at least one full business cycle?
5. Does the effect hold across key segments, or is it driven by one outlier cohort?

What is a p-value? Explain it to a non-technical PM. +
Trap: "Probability the result is due to chance"

That phrasing is technically wrong and interviewers at top companies will catch it.

Correct framing: "Imagine the new feature had zero real effect. The p-value tells us how often we would still see a difference this large just by random luck in who got assigned to each group. A p-value of 0.04 means: if nothing was actually different, we'd see this result by chance only 4% of the time. That's pretty unlikely, so we have evidence the feature is doing something real."

Why not just run the test until you see significance? +
Trap: "That's fine if you're careful"

It is not fine. Peeking and stopping at significance inflates your Type I error rate dramatically. If you check continuously and stop the moment p < 0.05, your true false positive rate can exceed 30%, not 5%.

The fix is either: (a) pre-register your sample size and only look once, (b) use sequential testing methods like alpha spending or always-valid p-values, or (c) apply a Bonferroni-style correction for multiple looks.

How do you handle network effects in an A/B test? +
Trap: Treating it as a standard user-level experiment

Network effects violate SUTVA. If User A is in treatment and interacts with User B in control (e.g., sends them a message using a new feature), the treatment "contaminates" the control group. Your estimated effect will be biased toward zero.

Solutions: Cluster randomization (randomize by social graph cluster, city, or marketplace region), ego-network experiments, or switchback designs for time-series spillovers. This is a core Uber, Airbnb, and LinkedIn interview topic.

What is a Sample Ratio Mismatch (SRM) and why does it matter? +
Trap: Proceeding with analysis despite SRM

SRM happens when the observed traffic split between control and treatment does not match the intended split. For example, you planned 50/50 but got 48/52.

Why it matters: SRM signals that randomization broke down. The groups may no longer be comparable, which invalidates all your metric comparisons. Common causes: bot traffic hitting one variant, caching issues, logging discrepancies, or a redirect bug affecting one arm.

Rule: If SRM is present, stop analysis, investigate the root cause, and rerun the experiment. Never rationalize through it.

When would you NOT run an A/B test? +
Trap: Saying "always A/B test everything"

Strong candidates demonstrate they know the boundaries of the tool.

Key scenarios: (1) Insufficient traffic -- you cannot reach power in a reasonable time. (2) Network effects make isolation impossible. (3) One-time irreversible changes like a full pricing model overhaul. (4) Ethical or legal constraints that prevent randomized denial. (5) The intervention is so large that a holdout group becomes indefensible.

Alternatives: Quasi-experiments, difference-in-differences, synthetic control, geo-based experiments.

Explain Type I vs Type II errors. Which is worse? +
Trap: Saying one is always worse

The right answer is: it depends on the product context.

Type I (false positive): You ship something with no real effect. Cost = wasted engineering, potentially degraded user experience, misleading future decisions.

Type II (false negative): You miss a real improvement and don't ship. Cost = lost growth, slower product iteration.

For safety-critical features, Type I is far worse. For a high-velocity growth team, Type II might be the bigger risk. Frame your answer around the business context.
